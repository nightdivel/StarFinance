# Star Finance - Техническое задание: раздел "Инструменты"

## 1. Цель
Создать раздел "Инструменты", в котором пользователь с правами управления может:
- добавлять кастомные функции в виде кнопок;
- задавать для кнопки название, описание и иконку (картинку);
- удалять ранее созданные кнопки;
- запускать функции из карточки инструмента.

Раздел должен закрывать операционные задачи без перехода в сторонние интерфейсы.

## 2. Scope
Входит в MVP:
- CRUD пользовательских инструментов (создание, просмотр, обновление, удаление).
- 4 типа действий:
  - переход по HTTP/HTTPS адресу;
  - запуск REST API функции;
  - отправка сообщения в группу Telegram;
  - отправка сообщения в канал Discord.
- Иконка инструмента: загрузка изображения или URL.
- Логи запусков (минимум статус, время, инициатор, сообщение ошибки).

Не входит в MVP:
- конструктор сложных мультишаговых workflow;
- планировщик по cron;
- визуальный редактор шаблонов с drag-and-drop.

## 3. Роли и права
Рекомендуемый новый ресурс прав: tools.

- tools:read
  - просмотр списка инструментов;
  - запуск разрешенных инструментов.
- tools:write
  - создание/редактирование/удаление инструмента;
  - управление видимостью и категориями.

Если в проекте пока нет ресурса tools, временно использовать settings:write для администрирования инструментов и settings:read для чтения.

## 4. Основные сценарии
UC-1. Создание инструмента
1. Пользователь открывает раздел "Инструменты".
2. Нажимает "Добавить инструмент".
3. Заполняет поля формы.
4. Сохраняет.
5. Новый инструмент появляется в каталоге и доступен к запуску согласно правам.

UC-2. Удаление инструмента
1. Пользователь открывает карточку/меню инструмента.
2. Нажимает "Удалить".
3. Подтверждает действие в модальном окне.
4. Инструмент скрывается из списка, фиксируется soft delete.

UC-3. Запуск инструмента
1. Пользователь нажимает "Запустить".
2. При необходимости вводит параметры (сообщение, переменные, payload).
3. Система выполняет действие и возвращает статус.
4. В UI отображается результат и запись в истории запусков.

## 5. Модель данных

### 5.1 Таблица tools
- id (uuid, pk)
- code (text, unique, системный идентификатор)
- title (text, 2-60)
- description (text, до 500)
- action_type (enum: open_url | rest_call | telegram_send | discord_send)
- icon_source (enum: upload | external_url)
- icon_url (text, nullable)
- icon_file_path (text, nullable)
- config_json (jsonb, обязательный для action_type)
- category (text, nullable, например "Операции", "Коммуникации")
- is_active (bool, default true)
- is_deleted (bool, default false)
- sort_order (int, default 100)
- created_by (int/uuid)
- updated_by (int/uuid)
- created_at (timestamptz)
- updated_at (timestamptz)

### 5.2 Таблица tool_runs
- id (uuid, pk)
- tool_id (uuid, fk -> tools.id)
- initiated_by (int/uuid)
- input_json (jsonb)
- output_json (jsonb)
- status (enum: queued | success | failed | blocked)
- error_message (text)
- started_at (timestamptz)
- finished_at (timestamptz)

### 5.3 Секреты и токены
Секреты не хранить в tools.config_json в открытом виде.
Использовать:
- переменные окружения;
- отдельное защищенное хранилище секретов;
- либо таблицу encrypted_secrets с серверным шифрованием.

## 6. Конфигурация по типам действий

### 6.1 open_url
config_json:
```json
{
  "url": "https://example.com/path",
  "openMode": "new_tab"
}
```
Правила:
- разрешены только http/https;
- запрет javascript: и data: схем;
- whitelist доменов (опционально для enterprise режима).

### 6.2 rest_call
config_json:
```json
{
  "method": "POST",
  "url": "https://api.example.com/v1/action",
  "headers": {
    "Content-Type": "application/json"
  },
  "auth": {
    "type": "bearer_env",
    "tokenEnv": "TOOLS_API_TOKEN"
  },
  "bodyTemplate": {
    "source": "starfinance",
    "user": "{{currentUser.username}}"
  },
  "timeoutMs": 10000
}
```
Правила:
- ограничить таймаут (например 10 секунд);
- ограничить размер body;
- retry не более 1-2 раз только для idempotent операций;
- логировать только безопасные поля (без токенов).

### 6.3 telegram_send
config_json:
```json
{
  "botTokenEnv": "TELEGRAM_BOT_TOKEN",
  "chatId": "-1001234567890",
  "messageTemplate": "{{currentUser.nickname}} запустил инструмент {{tool.title}}",
  "parseMode": "Markdown"
}
```
Правила:
- chatId валидировать;
- ограничить длину сообщения;
- экранировать markdown спецсимволы;
- защита от flood (rate limit).

### 6.4 discord_send
config_json:
```json
{
  "webhookUrlEnv": "DISCORD_TOOLS_WEBHOOK_URL",
  "messageTemplate": "Инструмент {{tool.title}} выполнен пользователем {{currentUser.username}}",
  "username": "StarFinance Bot"
}
```
Правила:
- хранить webhook только как secret/env;
- ограничить частоту отправки;
- разделять каналы по категориям уведомлений.

## 7. API контракт (предложение)

### 7.1 Инструменты
- GET /api/tools
  - права: tools:read
  - ответ: { items: Tool[] }

- POST /api/tools
  - права: tools:write
  - тело: CreateToolDto
  - ответ: { id }

- PUT /api/tools/:id
  - права: tools:write
  - тело: UpdateToolDto
  - ответ: { ok: true }

- DELETE /api/tools/:id
  - права: tools:write
  - soft delete
  - ответ: { ok: true }

- POST /api/tools/:id/run
  - права: tools:read
  - тело: { input?: object }
  - ответ: { runId, status }

### 7.2 История запусков
- GET /api/tools/runs
  - права: tools:read
  - фильтры: toolId, status, dateFrom, dateTo
  - ответ: { items: ToolRun[] }

## 8. UI/UX требования (Best Practices)

### 8.1 Информационная архитектура
- Вкладки:
  - Каталог (запуск инструментов);
  - Управление (CRUD, доступно tools:write);
  - История (логи запусков).
- Поиск и фильтры: по названию, категории, типу действия, активности.

### 8.2 Каталог инструментов
- Отображение плитками (card grid) + режим списка.
- В карточке:
  - иконка;
  - название;
  - краткое описание (2 строки, далее ...);
  - тип действия (badge);
  - кнопка "Запустить";
  - вторичное меню (редактировать/удалить) для tools:write.
- Состояния:
  - normal/hover/disabled/loading/success/error.

### 8.3 Форма создания/редактирования
- Пошаговый мастер (3 шага):
  - Основное (название, описание, иконка, категория);
  - Тип действия и параметры;
  - Предпросмотр и тестовый запуск.
- Валидация inline и перед submit.
- Сохранение черновика при закрытии модалки.

### 8.4 Удаление
- Только через подтверждение:
  - заголовок: "Удалить инструмент?";
  - предупреждение о невозможности запуска после удаления;
  - обязательный явный action "Удалить".

### 8.5 UX-паттерны безопасности
- Ясный уровень риска рядом с кнопкой запуска:
  - low: open_url;
  - medium: telegram/discord;
  - high: rest_call (если write/POST/DELETE).
- Для high-risk включить дополнительное подтверждение.
- Лимиты частоты запусков на пользователя.

### 8.6 Доступность
- Контраст не ниже WCAG AA.
- Полная клавиатурная навигация.
- visible focus для всех интерактивных элементов.
- alt-текст для иконок.
- Ошибки формы связаны с полями и озвучиваются screen reader.

### 8.7 Мобильный UX
- На ширине < 768px карточки в 1 колонку.
- FAB "Добавить" только для tools:write.
- Sticky фильтры и быстрый поиск.

## 9. Предложение полезных функций для пользователей
Дополнительно к 4 базовым типам действий:

1. Копирование данных в буфер обмена
- Быстро копировать ID заявки, ссылку, форматированный текст.

2. Открытие внутреннего раздела приложения
- Быстрый переход в конкретный модуль с предустановленным фильтром.

3. Генерация шаблонных сообщений
- Автосборка текста для Discord/Telegram из данных заявки/товара.

4. Webhook с подтверждением
- Отправка событий во внешние системы с preview payload.

5. Массовые действия по выбранным элементам
- Пример: отправить сводку по нескольким заявкам одним сообщением.

6. Экспорт в CSV/JSON
- Выгрузка текущего списка с учетом фильтров.

7. Инструмент "Проверка доступности"
- Ping выбранного API endpoint и показ latency/status.

8. Запуск по шаблону параметров
- Сохраненные пресеты input для часто повторяемых запусков.

## 10. Нефункциональные требования
- Производительность:
  - каталог инструментов до 1000 записей с пагинацией;
  - p95 ответа GET /api/tools <= 500ms.
- Надежность:
  - повторная отправка только для безопасных операций;
  - идемпотентный ключ для внешних вызовов (опционально).
- Аудит:
  - кто создал/изменил/удалил инструмент;
  - кто и когда запускал.

## 11. Критерии приемки
- Пользователь с tools:write может создать инструмент каждого из 4 типов.
- Пользователь с tools:read может запустить инструмент и увидеть статус.
- Удаление не удаляет историю запусков.
- Иконка отображается в карточке и в списке.
- Ошибки внешнего вызова показываются в UI без утечки секретов.
- Мобильная версия поддерживает создание и запуск без потери функциональности.

## 12. Этапы реализации
Этап 1 (MVP):
- backend CRUD + run endpoint + простые логи;
- frontend каталог + мастер создания + удаление;
- 4 action type.

Этап 2:
- пресеты параметров;
- расширенная история и фильтры;
- rate limit и риск-уровни в UI.

Этап 3:
- шаблоны с переменными из доменных сущностей;
- массовые действия;
- webhook-подписи и расширенный аудит.
